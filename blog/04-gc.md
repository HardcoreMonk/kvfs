# kvfs from scratch, Ep.4 — Rebalance가 남긴 과제, GC가 푸는 두 안전망

> Season 2 · Episode 3. Ep.3의 마지막 줄("over-replicated 보존, GC는 별도 ADR")을 메우는 episode. **claimed-set + min-age** 두 안전망 하나만으로 데이터 손실 위험 0인 채로 디스크와 메타를 정렬합니다.

## Ep.3가 남긴 빚 — 디스크와 메타의 불일치

`demo-eta.sh`의 마지막 출력은 솔직했습니다:

```
[7/7] Disk-level confirmation
  dn1  chunk count = 4
  dn2  chunk count = 4   ← 메타는 일부 청크에 대해 dn2 를 안 가리키는데...
  dn3  chunk count = 4
  dn4  chunk count = 3
```

15개 disk chunk · 12개 meta-claimed pair. **3개의 surplus.** ADR-010은 이 빚을 명시적으로 미뤘죠 — "GC는 별도 ADR감". 이번 episode가 그 ADR입니다.

## 만일 GC가 멍청하다면

가장 쉬운 GC: "메타가 가리키지 않는 디스크 청크를 즉시 삭제." 코드 5줄이면 됩니다. 그리고 거의 확실히 사고를 칩니다.

시나리오:
1. 운영자가 `rebalance` 직후 `gc`를 실행
2. GC가 dn2의 disk를 list — 청크 X 발견
3. 메타에서 X를 검색 — 한 객체의 replicas에 dn2 없음 → 삭제 결정
4. **그 사이**, 다른 객체가 PUT됨. 우연히 sha256 충돌 없이 같은 chunk_id가 만들어진다고 칩시다 (dedup 동작 — 같은 내용 PUT). 메타는 새 객체에 dn2 포함된 replicas를 적음
5. GC가 삭제. 새 객체의 한 replica가 사라짐

또는 더 흔한 race:
1. PUT이 DN 디스크에 청크 쓰기 직후, 메타 갱신 전 (수 ms 윈도우)
2. GC가 이 순간에 list → 메타 미반영 → 삭제

이런 race는 1년에 한 번이라도 일어나면 운영자 신뢰가 무너집니다.

## ADR-012의 답 — 두 안전망의 AND

```
삭제 후보 = 디스크에 있음 AND (chunk_id, addr) ∉ claimed-set AND mtime < now - min_age
```

두 안전망:

1. **claimed-set** — 메타 스냅샷에서 모든 (chunk_id, addr) 페어를 미리 모음. 하나라도 가리키면 삭제 금지. **dedup 친화적**: 같은 chunk_id가 여러 객체의 replicas에 등장하면 union 처리
2. **min-age** — 청크 mtime이 N초 이내면 삭제 금지. 기본 60초. PUT → 메타 갱신 propagation의 race window를 통째로 덮음

AND로 결합. 두 안전망 중 하나라도 보호하면 청크는 살아남습니다.

```go
// internal/gc/gc.go (핵심)
for _, addr := range coord.DNs() {
    chunks, _ := coord.ListChunks(ctx, addr)
    for _, c := range chunks {
        if claimed[c.ID][addr]    { continue }   // 안전망 1
        if c.MTime > cutoff       { continue }   // 안전망 2
        plan.Sweeps = append(plan.Sweeps, Sweep{addr, c.ID, c.Size, age})
    }
}
```

## Rebalance도 손봐야 했다

GC를 만들면서 ADR-010의 작은 결함이 드러났습니다. 기존 코드는 successful copies 후 메타를 `Actual ∪ Successful`로 갱신 — 즉 dn2를 surplus로 식별해놓고도 메타에 그대로 두었습니다. 결과:

- 메타가 항상 over-claim → GC가 식별할 surplus가 0
- ADR-010의 "메타도 desired에 맞춤" 약속과 불일치

수정 (`migrateOne`):

```go
// 모든 Missing 복사 성공 → 메타를 Desired로 trim (surplus DN을 메타에서 빼)
//                          (디스크는 손대지 않음 — GC의 일)
// 부분 실패 → 메타 = Actual ∪ Successful (안전망 유지)
if len(copyErrs) == 0 {
    finalReplicas = uniqueSorted(append(intersect(Actual, Desired), successfulMissing...))
} else {
    finalReplicas = uniqueSorted(append(Actual, successfulMissing...))
}
```

부분 실패 분기가 핵심. 모든 dst가 성공한 경우에만 surplus를 메타에서 뺍니다. 한 dst라도 실패하면 surplus를 메타에 유지해 R replicas 보장. **rebalance 자신의 데이터 손실 위험 0 약속을 깨지 않으면서** GC 가 일할 자료를 만들어냄.

## DN에 단 하나의 새 엔드포인트

```
GET /chunks → {"dn":"dn2","count":42,"chunks":[{id, size, mtime}, ...]}
```

`internal/dn/dn.go`에 ~30줄 추가. `filepath.Walk`로 디스크 enumerate, mtime은 `os.FileInfo.ModTime()`. 청크 메타 별도 저장 안 함 — 디스크가 곧 진실.

GC 가 이걸 호출해 in-memory에서 claimed-set과 비교. O(전체 청크 × DN) 메모리. 청크 100만 개 × 10 DN = 1000만 엔트리 hash map → 수백 MB. MVP 한계, 페이징은 ADR-013+ 후보.

## 라이브 데모 — `demo-theta.sh`

`./scripts/demo-theta.sh`는 down → up → 4 seed → dn4 추가 → rebalance → gc 를 한 흐름으로 시연:

```
=== θ demo: surplus-chunk GC (ADR-012) ===

[4/8] Run rebalance (ADR-010): copy misplaced chunks to dn4
⚙️  Rebalance applied (concurrency=4)
   scanned:      4
   planned:      3
   migrated:     3
   failed:       0
   bytes_copied: 114

  Disk after rebalance (note over-replicated dn1/2/3):
  dn1  chunk count = 4
  dn2  chunk count = 4
  dn3  chunk count = 4
  dn4  chunk count = 3
```

15개 디스크 청크 — 위 4 lines를 더하면 15. 메타는 4 객체 × 3 replicas = 12개 페어를 claim. 차이 3 = surplus.

```
[5/8] GC plan with min-age=5s (real-world default is 60s, demo uses short window)
🧹 GC plan — scanned 15 on-disk chunks across all DNs
   meta-claimed pairs (protected): 12
   min-age threshold (protected if newer): 5s
   sweeps proposed: 3

  dn2:8080  chunk=673a4bd3320a3077..  size=38  age=8s
  dn2:8080  chunk=f460f7849848bac9..  size=38  age=8s
  dn3:8080  chunk=33310bd95b6c23a8..  size=38  age=8s
```

**3개의 sweep**, 각 청크가 어느 DN의 디스크에 있는지·나이·크기까지 표시. 운영자가 plan을 보고 결정하기 좋은 형식.

```
[6/8] GC apply
🧹 GC applied (concurrency=4, min-age=5s)
   scanned:      15
   claimed:      12
   planned:      3
   deleted:      3
   failed:       0
   bytes_freed:  114

[7/8] Verify zero sweeps remain (idempotency)
🧹 GC plan — scanned 12 on-disk chunks across all DNs
   meta-claimed pairs (protected): 12
   min-age threshold (protected if newer): 5s
   sweeps proposed: 0
   ✅ No surplus to clean. Cluster disk = meta intent.

[8/8] Final disk state — should match meta intent exactly
  dn1  chunk count = 4
  dn2  chunk count = 2
  dn3  chunk count = 3
  dn4  chunk count = 3
```

세 가지 신호:

1. **planned 3 → deleted 3, failed 0, 114 bytes freed** — 정확한 매칭
2. **두 번째 plan = 0** — 멱등성. 같은 명령 100번 더 돌려도 0
3. **최종 disk count = 12 = 4 객체 × 3 replicas** — 메타 의도와 디스크가 정확히 일치. dn2는 2 청크만 (surplus 2개 삭제됨), dn3는 3 청크 (surplus 1개 삭제), dn1/dn4는 변화 없음

이 세 신호가 ADR-012의 약속을 그대로 시연합니다.

## edge에 추가된 단 두 개의 핸들러

```
POST /v1/admin/gc/plan?min_age_seconds=N         ← 안전, 동시 호출 OK
POST /v1/admin/gc/apply?min_age_seconds=N&...    ← s.gcMu 로 single-flight
```

Rebalance와 동일한 패턴. `apply`는 mutex로 직렬화. `plan`은 read-only로 동시 호출 가능.

## CLI — `kvfs-cli gc`

```
$ kvfs-cli gc --plan                    # 미리보기 (기본 min-age=60s)
$ kvfs-cli gc --plan --min-age 0 -v     # 즉시 모든 surplus 표시 (위험)
$ kvfs-cli gc --apply --concurrency 8
```

UX는 `kvfs-cli rebalance`와 정확히 동일 — 운영자가 한 가지 멘탈 모델로 둘 다 다룸.

## ADR-010 + ADR-012의 관계 — 직교성

| 책임 | rebalance | gc |
|---|---|---|
| 동기화 방향 | **메타 → 디스크** (없는 곳에 copy) | **디스크 → 메타** (메타가 모르는 거 삭제) |
| 메타 변경 | `Replicas` 갱신 (full success 시 trim) | 안 함 |
| 디스크 변경 | PUT 만 (never delete) | DELETE 만 (never copy) |
| 안전 규칙 | copy-then-update + (full success 시만 trim) | claimed-set ∧ min-age |
| 동시 실행 | 안전 (직교) | 안전 (직교) |

**서로의 영역에 안 들어감.** 한 ADR이 한 약속을 지키고, 두 ADR이 함께 클러스터 정렬을 완성.

## 솔직한 한계 — 다음 episode가 풀어야 할 것들

1. **메타 단일 진실 가정** — bbolt 메타가 손상되면 GC 가 의도치 않은 대량 삭제 가능. 운영자 실수도 동일 위험. 백업·HA → ADR-014 후보
2. **자동 트리거 없음** — DN 추가 후 운영자가 명시 실행 필요. 폴링·정책·알림 → ADR-013 후보
3. **scan 비용** — O(전체 청크) in-memory hash. 100만 청크 이상은 페이지네이션 필요
4. **min-age = 60s 기본값** — 한 가지 숫자로 모든 시나리오 커버 어려움. 운영 환경에 따라 5분, 1시간이 더 적절할 수도

이 한계들을 알면서 ADR-012로 묶지 않은 이유는 ADR-010과 동일:

> **하나의 결정 = 하나의 보장**

ADR-012는 "메타가 모르는 디스크 청크를 안전하게 삭제"만 약속합니다. 그 약속을 깨끗이 지키는 두 안전망이 위 알고리즘. 자동화·관찰성·메타 복구는 별개 약속.

- "scanned X, claimed Y, sweeps Z" 의 X = Y + Z 항상 성립
