# P9-03 Bucket And Object API Grill-Me

## Context

P9-02 added the S3 protocol foundation:

- `internal/s3api`
- SigV4 header verification
- S3 XML errors
- S3 operation classification
- edge S3 catch-all routing
- smoke-suite skeleton

P9-03 is the first successful S3-compatible data-plane slice:

- CreateBucket, ListBuckets, DeleteBucket
- PutObject, GetObject, HeadObject, DeleteObject
- ListObjectsV2

The existing native `/v1/o` path already performs chunk writes, reads, HEAD,
DELETE, and JSON list operations. The new S3 front door must not leak native
JSON/text responses or break demo compatibility.

## Question

Should P9-03 implement S3 bucket/object success by calling the existing native
HTTP handlers internally, or should edge keep a separate S3 dispatch path that
reuses lower-level helper functions and emits S3-specific status, headers, XML,
and errors directly?

## Recommended Answer

Use a separate S3 dispatch path that reuses lower-level edge helpers, not the
native HTTP handlers.

Reason: native handlers are intentionally shaped for demos and `/v1/*` clients:
UrlKey auth, JSON responses, native status text, native list shape, and
compatibility shortcuts. S3 needs SigV4 auth, XML success/error responses,
S3-specific headers, idempotent delete semantics, bucket-existence checks, and
different list pagination. Calling native handlers internally would couple two
public contracts and make it easy for one surface to regress the other.

The implementation should instead:

- keep `internal/s3api` protocol-only;
- extract or reuse edge helper functions for metadata lookup, chunk write,
  chunk read, and delete;
- add bucket registry operations in `internal/store` and `internal/coord`;
- extend `edge.CoordClient` for bucket RPCs;
- keep native `/v1/*` behavior unchanged.

## Decision

Use the recommended separate S3 dispatch path.

P9-03 may refactor edge internals to share helper functions between native and
S3 paths, but it must not proxy S3 requests through native HTTP handlers. S3
responses must be generated by S3-specific code so AWS CLI and MinIO client
compatibility can evolve without forcing native API changes.

## Residual Risk

The separate S3 dispatch path duplicates some object orchestration decisions.
The implementation plan should reduce that risk by extracting small helpers
around chunk write/read/delete and metadata operations, while leaving response
rendering separate for each public surface.
